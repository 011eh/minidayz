## 1. 与现有代码的关系

- **无现成进度系统**：`Progression` 是全新 autoload，不改动现有脚本。
- **天赋效果消费方已部分有字段**，本系统只负责"持有 + 暴露查询"，落地在各消费方：
  - `status/character_status.gd` 有 `speed`（默认在 `player_status.gd._ready()` 设为 130）→ **runner 跑者**改这里。
  - `status/player_status.gd` 有 `health_restore_rate`(`NORMAL_RESTORE_RATE=0.46`)、`bleed_rate`(0.82)、`hunger`/`thirst`、`try_restore()`、`update_status()` → **regen 再生 / metabolism 代谢 / camel 骆驼 / hamster 仓鼠**改这里。
  - `character/player/inventory.gd` 装备/格子 → **trunkster 大背包**改这里。
- **关键原则（务必遵守）**：`Progression` **不主动伸手改别人**。它只暴露 `has_perk()` / `perk_modifier()`，由各消费系统在自己的结算点**主动拉取**。这样 06/08/04/10/inventory 与本系统**解耦**——本系统不 `import` 它们。

> ⚠️ 提示：原 prompt 提到的 `to_restore()` 在代码里实际是 `try_restore()`；回血字段是 `health_restore_rate`、出血字段是 `bleed_rate`，本文以代码实测为准。

## 2. 数值 / 规格（原版数据，✅ 事件表确认）

### 2.1 经验获取

**被动**（在 Map 场景、玩家存活时，每 **10s** 按当前**地图层级 CurrentLevel** 加经验，来源 ✅ 50158-50172）：

| 地图层级 CurrentLevel | 每 10s 经验 | 含义 |
|---:|:---:|---|
| 0 | +1 | 初始森林（地表起始群系） |
| 1 | +2 | 更深一层 |
| 2 | +3 | 更深 |
| 3 | +3 | 最深（地堡方向，封顶 +3） |

**主动**（✅ 已核）：
- **击杀僵尸：+8 XP**。击杀事件归属 **07 僵尸与生物AI**；其击杀路径调用本系统干净 API：`Progression.add_xp(8)`。
- 拾取/探索等其它主动来源原文档未穷举（待核 §7-③），先只接击杀。

**内部计数偏移**：经验存于内部 `xp:int`，**起算值 = 16**；**UI 显示值 = xp - 16**（原版 t543/t596 文本，每 0.3s 刷新）。即首 16 点是基线偏移，界面从 0 开始显示。

### 2.2 天赋花费曲线（perk_cost，✅ 28455）

每买一个天赋后重算下个价格，公式（`n` = **已学天赋数**）：

```
perk_cost = 300 * n + 216
```

| 第几个（买前 n） | perk_cost（内部 XP） |
|:---:|:---:|
| 第 1 个 (n=0) | 216 |
| 第 2 个 (n=1) | 516 |
| 第 3 个 (n=2) | 816 |
| 第 4 个 (n=3) | 1116 |
| 第 k 个 (n=k-1) | 300(k-1)+216，线性 +300/个 |

- **购买条件**：`xp ≥ perk_cost` → 扣 `xp -= perk_cost`、`learned_perks` 加入该 id、播 "MD_Perk_1" 音、`perk_learned` 信号。
- 公式内含的 -16 偏移与显示 -16 偏移共同对齐 16 基线（原版细节，本项目按上式直接实现即可，无需单独减 16）。

### 2.3 "等级"的定义（澄清）

原版**没有独立的玩家等级 tier**：所谓"等级"在语义上就是**已学天赋数 `perks_learnt_count`**（决定下个天赋价格）。
另有一个**地图层级 `CurrentLevel`**（群系/深度，0-3），它只用来决定**被动 XP 速率**，与天赋数无关。两者不要混淆：

| 概念 | 本文变量 | 作用 |
|---|---|---|
| 天赋数（≈玩家"等级"） | `learned_perks.size()` | 决定下个 `perk_cost` |
| 地图深度层级 | `map_level`(0-3) | 决定被动 XP 速率（§2.1） |

> `map_level` 的来源：关联**地堡深度 / 关卡**（cross-ref `01` 阶段 E 地堡、`12` 游戏模式）。本系统不自己算，向外部只读获取（见决策 D-层级）。

## 4. Godot 数据模型与节点结构

### 4.1 `Progression` 自动加载单例

```gdscript
# progression/progression.gd  （AutoLoad 名: Progression）
extends Node

signal xp_changed(display_xp: int)        # 发的是显示值 = xp-16
signal level_up(perk_count: int)           # 已学天赋数变化（"等级"提升）
signal perk_learned(perk_id: StringName)

const XP_BASELINE := 16                     # 内部起算偏移
const KILL_XP := 8
const PASSIVE_INTERVAL := 10.0              # s

var xp: int = XP_BASELINE                   # 内部值；显示 = xp - XP_BASELINE
var learned_perks: Dictionary = {}          # { &"runner": true, ... } 当 Set 用
var perk_db: Dictionary = {}                # { &"runner": PerkResource, ... }

var map_level: int = 0                       # 0-3，外部只读注入（决定被动速率）

func display_xp() -> int: return xp - XP_BASELINE

func add_xp(amount: int) -> void:            # ← 击杀路径(07)、被动 timer 都走这里
    xp += amount
    xp_changed.emit(display_xp())

func next_perk_cost() -> int:                # 300n + 216
    return 300 * learned_perks.size() + 216

func can_buy(perk_id: StringName) -> bool:
    return not learned_perks.has(perk_id) and xp >= next_perk_cost()

func buy_perk(perk_id: StringName) -> bool:
    if not can_buy(perk_id): return false
    xp -= next_perk_cost()                    # 先扣（cost 用扣前的 n）
    learned_perks[perk_id] = true
    xp_changed.emit(display_xp())
    perk_learned.emit(perk_id)
    level_up.emit(learned_perks.size())
    # play "MD_Perk_1"
    return true

# —— 对外只读契约（消费方主动拉取，禁止本类反向调用消费方）——
func has_perk(perk_id: StringName) -> bool:
    return learned_perks.has(perk_id)

func perk_modifier(key: StringName, default: Variant) -> Variant:
    # 聚合所有已学天赋中声明了该 key 的修正量（见 4.2 PerkResource.modifiers）
    var v: Variant = default
    for id in learned_perks:
        var r: PerkResource = perk_db[id]
        if r.modifiers.has(key):
            v = r.apply(key, v)               # 由 PerkResource 决定加/乘/置
    return v
```

被动 XP 由内部 `Timer`（`wait_time = PASSIVE_INTERVAL`）驱动：

```gdscript
const RATE_BY_LEVEL := [1, 2, 3, 3]          # index = map_level
func _on_passive_tick() -> void:
    if not _player_alive(): return            # 仅存活时
    add_xp(RATE_BY_LEVEL[clampi(map_level, 0, 3)])
```

> 存档/重开时 `xp` 复位到 `XP_BASELINE`、`learned_perks` 清空（原版 50258-50265 所有 perk 重置）。归 `12` 存档系统调 `Progression` 的 reset。

### 4.2 `PerkResource`（每天赋一个 .tres）

```gdscript
# progression/perk_resource.gd
@tool
class_name PerkResource extends Resource

@export var id: StringName                    # &"runner"
@export var perk_name: String                 # "跑者"
@export var frame: int                         # 0-15，面板图标帧
@export_multiline var description: String
# 效果以"可读修正量"声明，而非硬编码逻辑：
@export var flags: Array[StringName] = []      # 纯开关型，如 &"agronom_no_wilt"
@export var modifiers: Dictionary = {}         # { &"player_speed_walk": 110, ... }

func apply(key: StringName, current: Variant) -> Variant:
    # 默认"置为该值"；如需加/乘可在此按约定（如 {op="add", value=3}）扩展
    return modifiers[key]
```

- **资源目录**：`progression/perks/perk_runner.tres` … 共 16 个，启动时由 `Progression` 扫描装进 `perk_db`。
- **不复刻 Construct2 的 `iv[]` / 帧索引天赋编码**：原版用全局 `perk_<name>=0/1` + 菜单帧 index；本项目用 **id 字符串 + Resource + Dictionary 修正量**，语义自解释。
- **frame** 仅供 UI 取图标用，不承载逻辑。

### 4.3 节点结构（autoload + UI 面板）

```
Progression (Node)                         AutoLoad；持 xp/learned_perks/perk_db
└── PassiveTimer (Timer)                    wait_time=10，autostart，按 map_level 加 XP

PerkPanel (Control)                        天赋购买面板（菜单里打开）
├── XPLabel (Label)                         显示 display_xp() + next_perk_cost()
└── PerkGrid (GridContainer)                columns≈4，16 个格子（对应原版菜单帧布局）
    └── PerkButton (Button) ×16            icon=PerkResource.frame；
        每个绑定 perk_id；
        已学→选中态、买不起→disabled；
        按下→Progression.buy_perk(id)
```

- 面板订阅 `Progression.xp_changed / perk_learned / level_up` 刷新格子状态与价格文本。
- 原版购买菜单的 4×4 帧布局可直接复用为 `GridContainer(columns=4)`。

## 5. 天赋效果"可读修正量"契约（解耦核心）

各消费系统在自己的结算点**主动查询** `Progression`，本系统**绝不反向调用它们**。约定如下（key 名仅示意，落地时与各文档对齐）：

| 天赋 | 消费方查询点 | 查询方式 |
|---|---|---|
| runner | `player_status` / 移动脚本算 `speed` 时 | `speed = Progression.perk_modifier(&"player_speed_walk", 100)`（持枪态取 `..._gun`） |
| metabolism | `player_status.update_status()` 用 `bleed_rate` 前 | `var r = bleed_rate * (0.5 if Progression.has_perk(&"metabolism") else 1.0)` |
| regen | `player_status.try_restore()` | `if Progression.has_perk(&"regen") and health < 50: 额外回一次` |
| camel/hamster | 5s 状态结算 | `if Progression.has_perk(&"camel"): thirst += 0.3`（hamster→hunger） |
| trunkster | `inventory` 初始化格子数 | `slots = base + Progression.perk_modifier(&"backpack_extra", 0)` |
| farmer | `04` 收获产出 | `yield += Progression.perk_modifier(&"crop_yield_bonus", 0)`（+3） |
| agronom | `04`/`10` 枯萎检查 & 草药采集 | `if Progression.has_perk(&"agronom"): 跳过枯萎; 草药 ×2` |
| header / meleedef / meleeatk / hunter | `06` 命中/格挡/暴击/索敌范围 | `has_perk()` / `perk_modifier()` 取阈值与范围 |
| snowborn / sweethome | `08` 体温公式 | `temp += has_perk(&"snowborn") ? 2 : 0` 等 |
| eye | `08` 夜幕 overlay 透明度 | `alpha = has_perk(&"eye") ? 低值 : 高值` |
| spotter | `07`/HUD 敌人标记 | `if has_perk(&"spotter"): 显示敌人 marker` |

> 好处：新增/调整天赋只改 `.tres` + 对应消费点一行查询；本系统零改动，也不会因为缺少 06/07/08 而无法编译——查询方各自就绪时再接。
