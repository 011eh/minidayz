## 1. 数值/规格

### 1.1 难度档位常量（✅ 已核对原版）

三档难度 `Novice/Regular/Veteran`（新手/普通/老兵）。所有数值来自 `data.js` 全局变量定义段（`States` 组在游戏开始时赋值）。

**伤害类**（来源：`生物数值.md` §3.1，行 1383-1432）：

| 常量 | Novice | Regular | Veteran | 含义 |
|---|:--:|:--:|:--:|---|
| `zed_dmg`（普通/掘地/狼僵尸）| 6 | 9 | 10 | 路由 iv[2]=1/6 |
| `zed_army_dmg`（军队僵尸）| 9 | 13 | 15 | 路由 iv[2]=2 |
| `wolf_dmg`（狼）| 6 | 8 | 10 | — |
| `difficulty_damage_reducer` | 7 | 5 | 3 | 玩家承伤减算因子，**越大越抗揍** |
| `zed_fast_dmg`（快速僵尸）| 6 | 6 | 6 | **固定全局，不随难度变**（data.js 初值 = 6）|

**状态衰减类**（来源：`玩家状态衰减与体温.md` §6，每 5s 作用一次）：

| 常量 | Novice | Regular | Veteran | 含义 |
|---|:--:|:--:|:--:|---|
| `starving`（饥饿衰减 /5s）| 0.75 | 1.1 | 1.4 | — |
| `thirsty`（水分衰减 /5s）| 0.75 | 1.1 | 1.4 | — |
| `regeneration`（自然回血 /5s）| 2 | 1.75 | 1.5 | Novice 回血最快 |

## 2. GameConfig 自动加载（难度 + 模式权威源）

只读配置单例：开局确定，运行期一般不变（种子模式在新游戏菜单选定）。其它系统**只读**它，不各自存难度副本。

### 2.1 枚举

```gdscript
# global/game_config.gd —— 注册为自动加载 GameConfig
extends Node

enum Difficulty { NOVICE, REGULAR, VETERAN }

enum GameMode {
    NORMAL,        # 普通模式（SEED_RUN=0）
    GUNNER_ZEDS,   # 0 持枪僵尸（远程僵尸）
    VIP,           # 1 公敌：被通缉，敌人针对玩家
    MORE_ZEDS,     # 2 僵尸大增（且强制 rains_level=2）
    WILDLIFE,      # 3 野生动物激增
    STALKER,       # 4 潜行者：追踪型高威胁敌人
    PUBG,          # 5 大逃杀：PUBG loot 池 + 缩圈
    CASUAL,        # 6 休闲：不死 + 强制 Novice
    TARGET_DROP,   # 7 定向空投
}

signal difficulty_changed(value: Difficulty)
signal game_mode_changed(value: GameMode)

var difficulty: Difficulty = Difficulty.REGULAR :
    set(v): difficulty = v; difficulty_changed.emit(v)
var game_mode: GameMode = GameMode.NORMAL :
    set(v): game_mode = v; game_mode_changed.emit(v)

func is_seed_run() -> bool:
    return game_mode != GameMode.NORMAL
```

### 2.3 各系统读取约定（接口契约）

| 读取方 文档 | 读什么 | 用法 |
|---|---|---|
| **07 僵尸与生物AI** | `zed_dmg / zed_army_dmg / wolf_dmg / ZED_FAST_DMG / dmg_reducer`、`zed_per_base_*` | 命中玩家时按 iv[2] 取对应伤害；刷怪时取基值 |
| **08 体温昼夜天气** | `night_cold / rain_cold / day_cold` | 体温每跳寒冷扣减；CASUAL 强制 Novice、MORE_ZEDS 强制 rains_level=2 由模式钩子处理 |
| **状态系统**（`player_status.gd`）| `starving_5s / thirsty_5s / regen_5s`（折算到本项目 tick）| 见 §3 折算约定 |
| **05 战利品刷新系统** | `is_seed_run()` + `game_mode`（PUBG/TD）| PUBG 切 PUBG 池、TD 改空投 |

> 折算约定（避免 5s/1s 混淆）：`GameConfig` 额外暴露便捷方法把 /5s 折到本项目的 `STATUS_UPDATE_INTERVAL`：
> ```gdscript
> func hunger_rate_per_tick() -> float:
>     return diff("starving_5s") / 5.0 * PlayerStatus.STATUS_UPDATE_INTERVAL
> ```
> 状态系统在难度确定/变化时把 `PlayerStatus.hunger_decrease_rate` 等设为该值，而非沿用硬编码 0.24。

## 4. 8 种子模式（SEED RUNS）规则表

`is_seed_run()==true` 时，按 `game_mode` 改写对应系统行为。**8 模式互斥**（同一时刻仅一个）。**所有种子模式下角色/天赋解锁禁用**（仅 `NORMAL` 解锁，见 §6）。

| 模式 | 原版标志 / 帧 | 规则改写 | 触及系统 |
|---|---|---|---|
| `GUNNER_ZEDS` | SEED_gunnerzeds / 0 | 僵尸持枪，具备远程攻击 | 07 刷怪+僵尸行为 |
| `VIP` | SEED_VIP / 1 | 玩家被通缉为公敌，敌人主动针对/聚焦玩家 | 07 AI 仇恨 |
| `MORE_ZEDS` | SEED_morezeds / 2 | 僵尸密度大增；**强制 `rains_level=2`** | 07 刷怪密度、08 天气 |
| `WILDLIFE` | SEED_Wildlife / 3 | 野生动物（狼/熊等）激增 | 07 生物刷怪 |
| `STALKER` | SEED_stalker / 4 | 潜行者：高威胁追踪型敌人 | 07 特殊敌人 |
| `PUBG` | SEED_pubg / 5 | loot 改用 **PUBG 专用池**；缩圈玩法（圈外持续伤害、圈逐步收缩）| 05 loot 池、新增缩圈控制器、状态扣血 |
| `CASUAL` | SEED_playcasual / 6 | 生命 ≤0 不死（强制拉回 1）；**强制 Novice 难度** | 状态系统（死亡判定）、GameConfig 难度 |
| `TARGET_DROP` | SEED_TD / 7 | 空投按 iv[0] 投**定向武器组**（按武器类别）| 05 空投系统 |

实现挂钩思路（Godot 风）：各系统在自身入口查 `GameConfig.game_mode`，命中则走分支；缩圈/公敌等独有玩法做成**仅在该模式下实例化的子节点/控制器**，普通模式不加载。

> ⚠️ **各模式内部精确数值**（刷怪倍率、缩圈半径/速度、公敌追踪范围等）原文档**未逐组解析**（`角色解锁与游戏模式.md` §待核实 3）。首轮按「主题正确 + 接钩子」实现，精确数值待按需深挖各 SEED 子组（见决策「种子模式首轮范围」）。

## 5. SaveSystem 自动加载（存档持久化）

### 5.1 设计原则

- **位置**：`user://`（Godot 跨平台用户数据目录），文件名如 `user://save_slot_0.json`。
- **格式（推荐）**：**JSON via `FileAccess`**。理由：人类可读便于调试、版本演进容易加 `version` 字段、与「per-system `to_dict()`」契约天然契合。备选 `.tres` Resource 见决策。
- **契约**：每个有状态的系统实现 `to_dict() -> Dictionary` / `from_dict(d: Dictionary) -> void`（或 `save_state()/load_state()`）。`SaveSystem` 只负责聚合、写盘、读盘、分发，不懂各系统内部。
- **世界省存**：地图**程序生成**（01）——只存 **seed + 关卡/depth**，不存每个 tile。但**被玩家改动过的世界对象**（已搜刮的 loot 点、已部署的建造物、车辆状态）必须显式存状态。
- **瞬态不存**：生物（僵尸/动物）状态瞬态——**读档后按规则重新刷怪**，不逐只序列化。

### 5.2 存档 schema（聚合各系统 `to_dict()`）

### 5.3 物品序列化（依赖现有 ItemCreator）

物品是 Resource 支撑的（`item/item_creator.gd`）。**只存 `id + count + durability` 就够**——读档时用 `ItemCreator.create_item_from_id(id)` 重建，再回填 count/durability。无需序列化整个 Resource。

```gdscript
# 存：item -> {"id": item.resource.id, "count": item.count, "durability": item.durability}
# 读：var item = ItemCreator.create_item_from_id(d["id"]); item.count = d["count"]; ...
```

### 5.4 SaveSystem 骨架

```gdscript
# global/save_system.gd —— 注册为自动加载 SaveSystem
extends Node

const SAVE_PATH := "user://save_slot_0.json"
const SAVE_VERSION := 1

signal saved
signal loaded

func save_game() -> void:
    var data := {
        "version": SAVE_VERSION,
        "config": {"difficulty": GameConfig.difficulty, "game_mode": GameConfig.game_mode},
        "stats": Stats.to_dict(),
        "player_status": PlayerStatus.to_dict(),
        "inventory": _player_inventory().to_dict(),
        "progression": Progression.to_dict(),
        "world": World.to_dict(),     # 聚合 01/05/08/10/11
    }
    var f := FileAccess.open(SAVE_PATH, FileAccess.WRITE)
    f.store_string(JSON.stringify(data, "\t"))
    f.close()
    saved.emit()

func has_save() -> bool:
    return FileAccess.file_exists(SAVE_PATH)

func load_game() -> void:
    if not has_save(): return
    var f := FileAccess.open(SAVE_PATH, FileAccess.READ)
    var data: Dictionary = JSON.parse_string(f.get_as_text())
    f.close()
    # 1) 先恢复 config / stats（影响后续生成）
    GameConfig.difficulty = data["config"]["difficulty"]
    GameConfig.game_mode  = data["config"]["game_mode"]
    Stats.from_dict(data["stats"])
    # 2) 用 seed 重建世界，再覆盖被改动的对象状态
    World.from_dict(data["world"])     # 内部：先按 map_seed 生成，再套 looted/deployed/vehicles
    # 3) 玩家状态/背包/进度
    PlayerStatus.from_dict(data["player_status"])
    _player_inventory().from_dict(data["inventory"])
    Progression.from_dict(data["progression"])
    loaded.emit()
```

> 读档顺序关键：**config/stats → 世界(种子重建+覆盖) → 玩家/背包/进度**。先有难度与种子，世界才能正确重建；玩家最后落位。

### 5.5 自动存档节奏

- **退出存档**：`get_tree().auto_accept_quit = false` + `_notification(NOTIFICATION_WM_CLOSE_REQUEST)` 里 `save_game()` 后再退出（保证关窗不丢档）。
- **定时自动存档**：每 N 秒（建议 60s）或关键事件（进入新群系、换天）触发一次 `save_game()`。
- **首轮**：单槽（`save_slot_0`），覆盖式。多槽见决策。

### 6.3 角色解锁判定（CharacterRoster）

```gdscript
# global/character_roster.gd —— 自动加载 CharacterRoster
extends Node

signal character_unlocked(id: int)

var unlocked := {}   # {int: bool}，1-3 默认 true；存档持久化

func _check_unlocks() -> void:
    if GameConfig.is_seed_run():
        return   # 种子模式禁用解锁
    _try(4,  Stats.zeds_killed >= 5)
    _try(11, Stats.days_survived >= 8 and GameConfig.difficulty == GameConfig.Difficulty.VETERAN)
    _try(14, Stats.karma_points <= -111 or Stats.karma_points >= 89)
    # … 其余按 6.2 表，带难度门槛
func _try(id: int, cond: bool) -> void:
    if cond and not unlocked.get(id, false):
        unlocked[id] = true
        character_unlocked.emit(id)
```

`unlocked` 与 §7 的 `unlock_progress` 一并进 `Stats`/存档。各达成事件（击杀、止血、换天）由对应系统调 `Stats.*` 增量后触发 `_check_unlocks()`。

## 7. Stats 轻量计数器

驱动解锁与统计的小单例。07（击杀）、08/时间（天数）、06/NPC（karma）、状态系统（止血次数）等向它增量上报。

```gdscript
# global/stats.gd —— 自动加载 Stats
extends Node

signal stat_changed(key: String, value: int)

var zeds_killed := 0
var fist_kills := 0
var bandit_kills := 0
var magnum_bandit_kills := 0
var bleeds_stopped := 0
var days_survived := 0
var karma_points := 0          # 钳制 [-111, 89]
var unlock_progress := {}      # {char_id: progress}

func add_kill(by_fist := false) -> void:
    zeds_killed += 1
    if by_fist: fist_kills += 1
    stat_changed.emit("zeds_killed", zeds_killed)
    CharacterRoster._check_unlocks()

func add_karma(delta: int) -> void:
    karma_points = clampi(karma_points + delta, -111, 89)
    CharacterRoster._check_unlocks()

func to_dict() -> Dictionary: ...   # 见 §5.2 stats 段
func from_dict(d) -> void: ...
```

| 上报方 | 调用 |
|---|---|
| 07 僵尸死亡 | `Stats.add_kill(by_fist)`；按武器/目标细分 bandit/magnum |
| 状态系统 止血成功 | `Stats.bleeds_stopped += 1` |
| 08/时间 跨天 | `Stats.days_survived += 1` |
| 06/NPC 善恶行为 | `Stats.add_karma(±delta)`（delta 表待定，见 §1.2）|

## 10. 决策

### 已定 ✅
1. ✅ 不照搬原版 `iv[]`/`Stats_*`/WebStorage 平铺结构——改用「自动加载单例 + 枚举 + per-system `to_dict()/from_dict()` + JSON 写 user://」。
2. ✅ 难度/模式的**单一权威源**是 `GameConfig`，其它文档（07/08/05/状态）只读不重定义。
3. ✅ 世界**只存 seed + 关卡 + 被玩家改动的对象**（looted/deployed/vehicles），其余按种子重建；**生物瞬态不存，读档重刷**。
4. ✅ 物品只存 `id + count + durability`，读档用 `ItemCreator.create_item_from_id` 重建。
5. ✅ 难度数值照搬原版（§1.1，已核对来源行），首轮即按表实现。

### 待定 ⏸️
1. ⏸️ **存档格式 JSON vs Resource(.tres)**：当前倾向 **JSON**（可读、易版本演进）；若后续要强类型/防篡改可换 `.tres`（`ResourceSaver.save`）。schema 与契约对两者通用，切换成本低。
2. ⏸️ **多存档槽**：首轮**单槽**（`save_slot_0`，覆盖式）；多槽留接口（路径已参数化为 `save_slot_N`）。
3. ⏸️ **Karma 来源清晰度**：原文档未列出各动作的 karma 增量（§1.2）；`add_karma(delta)` 入口已留，**delta 表待 06 战斗/NPC 系统落地时核实补全**。
4. ⏸️ **种子模式首轮范围**：首轮做到「8 模式可选 + 主题正确 + 接钩子（解锁禁用、CASUAL 不死/强 Novice、MORE_ZEDS 强雨、PUBG 切 loot 池）」；各模式**精确数值**（缩圈半径/速度、刷怪倍率、公敌追踪范围）原文档未逐组解析，按需深挖各 SEED 子组后再补。
