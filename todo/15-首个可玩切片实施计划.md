# 首个可玩切片实施计划（M1「能搜刮」+ M2「能战斗」）

> 定位：本文是**跨文档关键路径**——把分散在 `05`/`06`/`07`/`02`/`03`/`03a` 的 `- [ ]` 清单，按**依赖顺序**收敛成一条 file-level 构建序列，交付 `00-总览路线图.md` §4 的两个里程碑：
> - **M1「能搜刮」**：进村 → 走到一个可搜刮容器 → 开箱 → 一把枪 `Item` 落地 → 拾进背包。（依赖 `05` + `02/03`）
> - **M2「能战斗」**：一只普通僵尸会追、会打玩家；玩家能射击/近战把它打死（掉 XP 桩），僵尸也能把玩家打掉血。（依赖 `06` + `07`）
>
> 各分文档讲「怎么设计」，本文只讲「先做哪个文件、做完怎么验」。每步给：(a) 做什么、(b) 改/建哪些文件、(c) 实现哪份设计文档的哪节、(d) 一行 **验收标准**（尽量落到 `test/` 下的测试场景）。
> **接口以 `00a-系统接口契约.md` 为唯一真相源**——`take_damage(DamageInfo)`、碰撞层 3/4、`LootSystem` 池号、发物品链路均按 `00a` 写，分文档与 `00a` 冲突时以 `00a` 为准。
>
> 关键路径（粗箭头 = 阻塞）：
> ```
> P0 前置代码修复 ──→ P1 共享伤害契约 ──┬─→ P2 M1 搜刮链
>                                       └─→ P3 M2 战斗链 ──→ P4 切片整合测试
> ```
> 说明：P2（搜刮）与 P3（战斗）在 P1 之后**逻辑上可并行**，但二者都硬依赖 P0 的 `ItemCreator` 手枪段修复（搜刮要刷枪、战斗要发枪给玩家）。建议先 P2 后 P3，因为 P3 的玩家受击实现复用 P1 的 `DamageInfo`，而搜刮链更短、更快见效。

## Phase 1 — 共享伤害契约脚手架（`00a §1` 的唯一耦合面）

> M1 不需要伤害，但 M2 的两端（玩家被僵尸打、僵尸被玩家打）都要 `take_damage(DamageInfo)` + 碰撞层 3/4。本阶段把**契约本体**先立起来，让 P3 的两端各自实现 `take_damage` 时有共同类型可用。

- [ ] **P1-1｜定义 `DamageInfo`**
  - 做什么：按 `00a §1` 签名建轻量载荷类：`enum Source { BULLET, MELEE, ZOMBIE, TRAP, FIRE, COLLISION }`，字段 `amount:float`、`source:Source`、`attacker:Node=null`、`is_crit:bool=false`。攻击方先 roll ±25% 再塞 `amount`。
  - 文件：新建 `combat/damage_info.gd`（`class_name DamageInfo extends RefCounted`）。
  - 实现：`00a §1` / `06 §2.1`。
  - **验收标准**：`DamageInfo.new(10, DamageInfo.Source.ZOMBIE, self)` 可构造，字段读取正确。

- [ ] **P1-2｜`PlayerStatus.take_damage(DamageInfo)`（玩家受击端）**
  - 做什么：在 `PlayerStatus`（autoload）实现 `take_damage(info: DamageInfo) -> void`：先按 `00a §1`「单参恒为 DamageInfo」约定接收；本切片**先做最小减伤**——`health -= max(5, info.amount)`（护甲/格挡/难度因子留 `06 §6.1`，标后续），存活时按 `info.source` roll 出血置 `is_bleeding`（`06 §6.2`：BULLET 50% / MELEE 25% / ZOMBIE ≈16.7%）。出血通路已通（`update_status()` 内 `bleed_rate=0.82`），只需置位。

  - 实现：`00a §1` / `06 §2.2 / §6.1（简化）/ §6.2`。
  - **验收标准**：测试场景里 `PlayerStatus.take_damage(DamageInfo.new(20, ZOMBIE, null))` 后，`PlayerStatus.health` 下降对应值（`died` 信号在归零时触发）。

## 明确不在本切片范围（边界，防止 scope 蔓延）

以下系统**本切片一律不做**，仅为各分文档的关键路径留好接口/桩，后续按 `00` 路线图 M3+ 推进：

- **昼夜 / 天气 / 体温**（`08`）：`TimeSystem`/`WeatherSystem`/`HeatSource`、`survival_seconds` 里程碑——全部不做（`00a §2/§8`）。
- **经验 / 等级 / 天赋**（`09`）：`Progression` autoload 不实装；击杀 XP 先打印桩，不接 `add_xp`/`perk_modifier`（`00a §3`、`07 §8`）。
- **难度配置**（`12 GameConfig`）：`take_damage` 减伤因子、僵尸伤害/刷新数缩放——先用 Regular 常量（玩家减伤简化为 `max(5,amount)`、僵尸伤害 `Zed_dmg=9`），不读 `GameConfig.scale`（`00a §4`）。
- **`LootSystem` 完整刷新**（`05 P0/P1`）：1480s 周期重刷、>1000px 重刷 / ≥1500px 回收、弹药盒充能（≥500px）、空投分层表、PUBG 变体——全部留后续；本切片只做「首开即时出货 + 单池」。
- **6 类僵尸 / 刷新器 / 噪声 / 寻路网格 / Boss / 动物**（`07 P1+`）：army/fast/shooter/buried/screamer、`ZedSpawnPoint`（60s / 1500–2000px）、`NoiseBus` 招引、NavigationServer2D 全图烘焙、坦克、狼/鹿/兔/乌鸦——均不做；追击用简化朝向移动。
- **战斗手感细节**（`06 P1+`）：动态散布 5 参数模型、近战格挡 `block_chance`、霰弹多弹丸、全自动/栓动节奏区分、爆头 ×3、枪口闪光/tracer/音效、空仓自动换弹、弓弩投射物——本切片散布用固定值、单发逻辑命中即可。
- **完整城镇生成**（`03/03a` 整村 + 道路 + 城市/军营/医院）：本切片容器用**手摆测试场景**；村庄 9 仅作 M1 载体的可选升级，不要求全 25 地点落子 / 道路对象 / Footprint terrain 全量。
- **死亡掉落池**（`07 §1.6` 直接物品 id 列表）：僵尸死亡先只发 XP 桩，不掉物品（`07 P2-3`）。
- **存档**（`12`）：`to_dict/from_dict` 不实现（`00a §6`）。
- **耐久归 0 后果 / 衣物受击磨损联动保暖**（`06 §7.1 P2 / §7.2`）：本切片武器磨损仅扣值，衣物磨损/保暖联动不做。

## 决策 · 待定（本切片需拍板或随实现确认）

> 多数沿用各分文档「已定」结论；以下是本切片新增或需在动手时确认的点。

- [x] **D-手枪 ID 边界**：已核对 `item/pistol.gd` 的 `RES_TABLE` 键 = 190..201，定 `range(190, 202)`（裁决 `05 §10 D-手枪边界`）。
- [x] **D-M1 容器载体**：本切片用**手摆测试场景**（`test/loot_test.tscn` + 一个 `SearchableContainer`），不阻塞于 `03` 整村生成；若 `03` 村庄 9 已就绪可选用真村，非必需（呼应 `03a §9 D2/D6`：首版村庄只摆建筑）。
- [x] **D-玩家受击体归属**：沿用 `06 §10 D-受击体归属`——僵尸近战用 1s 重叠结算直接调 `Player.take_damage`，玩家**不挂 Hurtbox**；远程射手（本切片无）才需玩家受击层。
- [x] **D-僵尸寻路**：本切片 `Chase` 用**简化朝向 `move_and_slide`**（直奔玩家，撞墙靠碰撞滑动），NavigationServer2D 全图烘焙留 `07 P0-5`/后续。
- [ ] **D-难度常量取值**：玩家减伤公式与僵尸伤害本切片用 Regular 档（减伤简化 `max(5,amount)`、`Zed_dmg=9`）——确认是否就用 9，还是先用一个更低的体感值便于测试（`GameConfig` 实装后统一读 `00a §4`）。
- [ ] **D-XP 桩形态**：击杀 +8 XP 先「打印日志」还是先建一个最小 `Progression` 桩（仅累加 `xp` int）？倾向**先打印**，`09` 实装时替换（`00a §3`）。
