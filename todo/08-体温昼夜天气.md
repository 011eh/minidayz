# 昼夜 · 天气 · 体温（三系统耦合设计）

> 定位：三个系统天然耦合——**昼夜**决定 `Day_mode`（夜/昼），**天气**决定 `Rain`（雨/雪），二者一起喂进**体温公式**。故合为一文。
> 调研来源：`origin/others/文档/数值探索/昼夜与天气系统.md`、`origin/others/文档/数值探索/玩家状态衰减与体温.md`（原始工程 Mini DAYZ 1.4.1 / Construct 2，`game_events.decoded.txt` + `data.js` 初值）。
> 现有代码：`status/player_status.gd`（`PlayerStatus` 自动加载，已有 `temperature` 字段 + 1 秒 tick）、`status/character_status.gd`、`item/res/script/gear_resource.gd`（`heat` 字段）、`character/player/inventory.gd`（装备槽）。
> 已就绪闲置资源：`asset/images/others/night_overlay-sheet0.png`、`asset/media/environment/{night_loop,md_winter_night_amb,ambient,md_winter_amb,rain_loop,rainroof,blizzard}.ogg`。
> 交叉引用：`06`（衣物耐久受击衰减）、`10`（建造部署：篝火/帐篷热源）、`11`（载具：驾驶舱保暖）、`12`（游戏配置：难度）。

## 1. 数值 / 规格（原版还原）

### 1.1 时间换算（来源：昼夜与天气系统.md §1）

| 概念 | 数值 | 来源 |
|---|---|---|
| 推进步长 | 每 **1.5 现实秒** = +1 游戏分钟 | ✅ `Every(1.5)`，`Timer_ALL += 1` |
| 一天 | **1440 游戏分钟**（24 小时） | ✅ `Timer_ALL > 1440 → Day_survived+1, 归零` |

> 天数里程碑（原版有，本项目按需）：`Day_survived=8` 解锁角色 11；累计存活 2160/4320/6640 秒触发成就。本文只需暴露 `day_count` 与 `survival_seconds`，里程碑逻辑归其它系统消费。

### 1.2 天气节律（来源：昼夜与天气系统.md §3）

| 概念 | 数值 | 来源 |
|---|---|---|
| 天气判定周期 | **360 现实秒**（`Rain_speed`，循环） | ✅ data.js 初值 |
| 起雨概率 | 当前无雨 → `choose(1,2,3,4)==1` = **1/4（25%）** | ✅ §3 |
| 停雨 | 当前有雨 → 判定时直接停 | ✅ §3 |
| `rains_level` | 0=无 / 1=雨 / 2=暴风雪 | ✅ data.js 初值 0 |
| 自动升级 | 存活 ≥ **5000 秒** → `rains_level=2`（TriggerOnce） | ✅ §3 |
| 地堡内 | `inBunker=1` 时不起雨（本文先记录，地堡后做） | ✅ §3 |

雨/雪粒子参数（来源 §3「雨幕粒子」表，每 0.4 秒生成一批）：

| `rains_level` | 表现 | 粒子速度 | 缩放 | 落角 | 音效 |
|---|---|---|---|---|---|
| 1 普通雨 | 斜落如雨 | `random(700,1500)` | `choose(1,2)` | `random(80,110)°` | `rain_loop`（屋檐下 `rainroof`） |
| 2 暴风雪 | 飘落如雪 | `random(50,150)` | `choose(1,2,3)` | `random(80,110)°` | `blizzard`（无屋顶音） |

> 关键差异：**雨快（700–1500）/ 雪慢（50–150）**。本项目用一套粒子节点，按 `rains_level` 切换 `initial_velocity_min/max`、`scale_amount` 与贴图帧即可，不必两套节点。

### 1.3 体温条方向（**已核实，必读**）

来源：玩家状态衰减与体温.md §1 / §5 / §7。体温条 `iv[16]` 量程 **0..100**：

- **100 = 最暖端**；`Player_temperature>100`（即时体温过高）→ **中暑 Heatstroke**（暴血 + 出血）。
- **0 = 最冷端**；体温条 `iv[16]≤0` → **失温 Hypothermia**，并 `random(50)>28` 即 **22/50 概率**起感染。

> 结论：**数值越高越暖、越低越冷**。本项目 `PlayerStatus.temperature` 沿用同向（已有 `is_health()` 要求 `temperature>=50`，与原版「回血需 iv[16]≥50」一致 ✅）。

### 1.4 难度寒冷常量（来源：玩家状态衰减与体温.md §5）

| 难度 | `Night_cold` | `Rain_cold` | `Day_cold` |
|---|---|---|---|
| Novice（新手） | 1 | 1 | 6 |
| Regular（普通） | 2 | 2 | 7 |
| Veteran（老兵） | 3 | 3 | 8 |

来源：✅ `game_events` 1391-1428。难度由游戏配置提供（交叉引用 `12`）。

### 1.5 即时体温公式（来源：玩家状态衰减与体温.md §5，**逐项照搬**）

原版每次 `Update_Stats`：`Player_temperature = 0`，再逐项加减：

| 条件 | 体温变化 |
|---|---|
| 夜晚（`Day_mode=0`） | `- Night_cold` |
| 下雨（`Rain=1`）且不在屋檐下 | `- Rain_cold` |
| 站在篝火上 | `+ 15` |
| `perk_snowborn` 天赋 | `+ 2` |
| 在热源区内（室内/暖区 `t337`） | `+ heat`；若 `perk_sweethome` 再 `+ heat/2` |
| 在地堡内（`inBunker=1`） | `+ 5` |
| 在可驾驶载具内（有油未损毁） | `+ 4` |
| **基础日寒（恒定）** | `- (Day_cold - (头保暖 + 上衣保暖 + 裤保暖))` |

结算到体温条：`体温条 += Player_temperature / 2`（原版每 5 秒一次，§2）。

衣物保暖换算（§4）：每件 `当前保暖 = (满保暖/100) × 当前耐久`。本项目映射为
`有效保暖 = gear.heat × (gear.durability / 100)`（`durability` 受击衰减见交叉引用 `06`）。

> ⚠️ **tick 频率折算**：原版体温每 **5 秒**结算一次（States 主循环 `Every(5)`），而本项目 `PlayerStatus.update_status()` 每 **1 秒**跑一次（饥饿 0.24/s、水分 0.24/s 已在此）。为忠实还原速率，**把 5 秒的体温增量按 1/5 摊到每秒**：每秒 `temperature += Player_temperature / 2 / 5`（等价于原版 5 秒一次的同等斜率），避免另起一个 5 秒计时器与现有 tick 竞争。

## 2. TimeSystem（自动加载）

游戏时钟单例，唯一时间真相源。`project.godot [autoload]` 现有 `PlayerStatus` / `ItemActionTable`，在其后追加 `TimeSystem`、`WeatherSystem`。

### 2.1 节点结构

```
TimeSystem (Node)                       # autoload, 无场景，纯逻辑
└── TickTimer (Timer)                   # wait_time = 1.5, autostart, 循环
```

### 2.2 字段 / 信号 / 接口

```gdscript
extends Node                            # autoload: TimeSystem

signal minute_changed(total_minutes)    # 每 1.5s
signal hour_changed(hour)               # 整点
signal day_changed(day_count)           # 跨天（含「第 N 天」提示时机）
signal day_phase_changed(is_night)      # 昼↔夜 翻转（驱动视觉/音频/体温）

const REAL_SECONDS_PER_GAME_MINUTE := 1.5   # ✅ Every(1.5)
const MINUTES_PER_DAY := 1440               # ✅ 一天 1440 分
const NIGHT_END_HOUR := 6                    # 0–5 夜，6–23 昼 ✅

var total_minutes := 0      # 当天已过分钟（原 Timer_ALL）
var day_count := 0          # 已存活天数（原 Day_survived）
var survival_seconds := 0.0 # 累计存活现实秒（原 Stats_time_lived，喂 5000s/天气升级）

func is_night() -> bool: return hour() < NIGHT_END_HOUR
func hour() -> int:   return total_minutes / 60
func minute() -> int: return total_minutes % 60
```

推进逻辑（`TickTimer.timeout`）：

```
survival_seconds += 1.5
total_minutes += 1
若跨小时 → hour_changed; 若 hour() 跨过 6↔0 边界 → day_phase_changed(is_night())
若 total_minutes > 1440 → total_minutes = 0; day_count += 1; day_changed(day_count)
```

> 用整型分钟计数 + 取模做时/分，避免浮点累计漂移。`survival_seconds` 即原版 `Stats_time_lived`，被 WeatherSystem 的 5000s 升级和将来的天数成就共用。

## 3. 昼夜视觉与音频

### 3.1 夜幕：CanvasModulate（推荐）vs ColorRect 叠层

| 方案 | 机制 | 优点 | 缺点 |
|---|---|---|---|
| **CanvasModulate（推荐）** | 整个 `CanvasItem` 树乘以一个颜色，调暗即「夜色」 | 一个节点全局生效；不挡 UI（UI 在 `CanvasLayer` 上，不被 2D 世界的 CanvasModulate 影响）；Tween `color` 天然平滑 | 是「整体染色」，无法做局部光斑（篝火光圈需另用 `Light2D`） |
| ColorRect 全屏叠层 | 半透明黑 `ColorRect` + `night_overlay-sheet0.png` | 可叠贴图纹理（原版 `t231` 夜幕贴图风格） | 必须铺满相机、跟随相机移动；易误盖 UI；多一层绘制 |

**决策：用 `CanvasModulate`**（见 §7 已定）。原版夜幕「不透明度 100% / 0%」对应到 `CanvasModulate.color` 在「夜色（如 `Color(0.25,0.28,0.45)`）」与「白昼 `Color.WHITE`」之间插值。`night_overlay` 贴图作为**可选**氛围叠层留待美术微调，不作为主方案。

```
World (Node2D)
└── DayNightModulate (CanvasModulate)   # color: 白天 WHITE ↔ 夜晚 偏蓝暗色
```

监听 `TimeSystem.day_phase_changed`：用 `create_tween()` 把 `color` 渐变（黎明/黄昏过渡）。原版在 0 时与 5 时按分钟做 `abs(分钟)*1.69` 的渐变（纯视觉，§2），本项目用一次 Tween（如 3–5 秒）近似即可，不必逐分钟驱动。

> 地堡内 `inBunker=1` 原版不切昼夜光照——本项目可在进入地堡时临时把 `DayNightModulate.color` 锁为白昼，离开恢复（地堡系统后做时接入）。

### 3.2 夜间环境音交叉淡入淡出

```
AmbientAudio (Node)
├── DayAmbient   (AudioStreamPlayer)   # md_winter_amb.ogg（雪地日）/ ambient.ogg
└── NightAmbient (AudioStreamPlayer)   # md_winter_night_amb.ogg（雪地夜）/ night_loop.ogg
```

- 原版按 `CurrentLevel`：≥2 用 `night_loop`/`ambient`，否则雪地 `md_winter_night_amb`/`md_winter_amb`（昼夜与天气系统.md §2）。本项目起步只有雪地群系 → 默认用 `md_winter_night_amb` / `md_winter_amb`，关卡系统接入后再按 `CurrentLevel` 切。
- 翻夜/翻昼时对两个 `AudioStreamPlayer` 的 `volume_db` 用 Tween 反向插值（一个淡入、一个淡出）实现交叉淡化，监听 `day_phase_changed`。

## 4. WeatherSystem（自动加载）

### 4.1 节点结构

```
WeatherSystem (Node)                    # autoload, 逻辑层
└── WeatherTimer (Timer)                # wait_time = 360, autostart, 循环 ✅

# 视觉/音频层挂在世界/相机下（由 WeatherSystem 持有引用或经信号驱动）：
Camera2D
└── RainLayer (Node2D)                  # 跟随相机，铺满视口
    ├── Rain (GPUParticles2D)           # 按 rains_level 切雨/雪参数
    └── WeatherAudio (AudioStreamPlayer)# rain_loop / blizzard / rainroof
```

### 4.2 字段 / 信号 / 接口

```gdscript
extends Node                            # autoload: WeatherSystem

signal weather_changed(is_raining, rains_level)

const WEATHER_CHECK_PERIOD := 360.0     # ✅ Rain_speed
const RAIN_START_CHANCE := 0.25         # ✅ choose(1,2,3,4)==1
const BLIZZARD_UPGRADE_SECONDS := 5000  # ✅ 存活升级阈值

var is_raining := false                 # 原 Rain
var rains_level := 0                    # 0 无 / 1 雨 / 2 暴风雪 ✅（初值 0）

func is_raining_outside() -> bool       # 用于体温：Rain==1 且玩家不在屋檐下
```

判定逻辑（`WeatherTimer.timeout`，照搬 §3）：

```
if rains_level == 0: return                  # 完全无雨群系不判定
if is_raining:
    is_raining = false                       # 停雨：淡出粒子、停雨声
else:
    if 不在地堡 and randf() < 0.25:           # 25% 起雨
        is_raining = true                    # 起雨：建粒子(帧=rains_level-1)、播对应音
weather_changed.emit(is_raining, rains_level)
```

自动升级（监听 `TimeSystem`，TriggerOnce）：`survival_seconds >= 5000 → rains_level = 2`（之后起的都是暴风雪）。群系初值：森林/初始场景 `rains_level=1`，冬季/雪地 `=2`（§3，由地图初始化设定，交叉引用 `01`）。

### 4.3 粒子（GPUParticles2D，跟随相机）

- `RainLayer` 挂在 `Camera2D` 下（或每帧同步 `global_position = camera.global_position`），使雨/雪始终铺满视口、不随玩家走出画面。
- 用 `ParticleProcessMaterial`：方向 = 落角 80–110°（接近竖直略斜）；速度按 `rains_level` 切 `initial_velocity_min/max`（雨 700–1500 / 雪 50–150）；`scale_min/max` 对应 `choose(1,2)` / `choose(1,2,3)`；贴图帧 0=雨、1=雪。
- 选 **GPUParticles2D**（量大、纯视觉、无碰撞需求，GPU 更省）；若目标平台 GPU 粒子兼容性差，回退 `CPUParticles2D`（API 近似，参数同名）。

### 4.4 雨声 / 屋檐

- `rains_level==1` → `rain_loop`；`==2` → `blizzard`。
- 玩家在屋檐下（与 §5 体温的「屋檐」同一判定）：暂停 `rain_loop`，改播 `rainroof`（室内听屋顶雨声）；暴风雪无屋顶音。

## 5. 体温公式与结算（接入现有 tick）

### 5.1 不另起计时器——复用 `update_status()`

`PlayerStatus.update_status()` 每 1 秒已在跑（hunger/thirst 0.24/s、bleed 0.82/s）。体温结算**加进同一函数**，不新建 `Timer`。

### 5.2 即时体温计算 `compute_temperature_delta()`

按 §1.5 逐项实现（返回原版「即时体温 `Player_temperature`」）：

```gdscript
func compute_temperature_delta() -> float:
    var t := 0.0
    var diff := GameConfig.difficulty            # 交叉引用 12 → Night_cold/Rain_cold/Day_cold
    if TimeSystem.is_night():
        t -= diff.night_cold
    if WeatherSystem.is_raining_outside() and not player_under_eave():
        t -= diff.rain_cold
    t += nearby_heat_bonus()                      # 篝火+15 / 暖区+heat / 载具+4 / 地堡+5（§5.4）
    if has_perk("snowborn"): t += 2
    # 基础日寒（恒定）：日寒被三件衣物保暖之和抵消
    t -= (diff.day_cold - clothing_heat_sum())
    return t

func clothing_heat_sum() -> float:
    var sum := 0.0
    for type in [CLOTHES, PANTS, HELMET]:         # inventory.gd 三个保暖槽
        var gear = inventory.equipment_slots[type]
        if is_instance_valid(gear):
            sum += gear.resource.heat * (gear.durability / 100.0)   # §4 耐久折算
    return sum
```

> 原版保暖槽是头/上衣/裤三件（iv[28]/iv[27]/iv[29]）。本项目 `inventory.gd` 的 `CLOTHES`/`PANTS`/`HELMET` 对应之；`VEST`/`BACKPACK` 原版不计入保暖（若 `gear_resource` 给它们设了 `heat`，按是否复刻原版决定纳入——见 §7 待定）。

### 5.3 结算到体温条（摊到每秒）

在 `update_status()` 末尾追加：

```gdscript
var temp_delta := compute_temperature_delta()
temperature += temp_delta / 2.0 / 5.0    # 原版 /2 后每 5s 一次 → 摊成每秒（§1.5 折算）
# temperature 的 setter 已 clamp(0,100)，无需手动夹
on_temperature_extreme(temp_delta)
```

### 5.4 热源契约（供 10 / 11 注册）

体温公式需要「玩家附近的暖意来源」。定义**热源接口**，让篝火/帐篷（`10`）、载具驾驶舱（`11`）即插即用：

**方案 A（推荐，区域驱动）**：热源是带 `Area2D` 的节点，加入分组 `"heat_source"`，暴露 `heat_value: float`。玩家身上挂一个 `Area2D`（或查询重叠），`nearby_heat_bonus()` = 累加所有重叠热源的 `heat_value`：

```gdscript
# 热源节点契约（10/11 实现）
class_name HeatSource          # 挂 Area2D，加入组 "heat_source"
@export var heat_value: float  # 篝火=15；暖区=t337.iv[0]；地堡=5；载具=4
@export var is_eave: bool      # 是否提供屋檐（挡雨）——影响 Rain_cold 与雨声切换
```

```gdscript
func nearby_heat_bonus() -> float:
    var bonus := 0.0
    for area in player_heat_probe.get_overlapping_areas():
        if area.is_in_group("heat_source"):
            var h := area.heat_value
            if area is_campfire and has_perk("sweethome"): h += h / 2.0   # 暖区 sweethome 翻半
            bonus += h
    return bonus
```

**方案 B（信号注册）**：热源在 `_ready` 时 `HeatSourceRegistry.register(self)`，离开时注销；玩家每 tick 查表算距离。比 A 复杂，仅在 Area2D 重叠开销成为瓶颈时考虑。

> `player_under_eave()` 复用同一探测：附近存在 `is_eave==true` 的热源即视为在屋檐下（与 §4.4 雨声共用）。
> 10/11 只需让自己的篝火/帐篷/载具节点满足 `HeatSource` 契约（加组 + 设 `heat_value`/`is_eave`），**无需改 PlayerStatus**。

### 5.5 极端后果

```gdscript
func on_temperature_extreme(instant_temp: float) -> void:
    # 中暑：原版 Player_temperature > 100（即时体温过高，非体温条满）
    if instant_temp > 100:
        health = 5
        is_bleeding = true                      # 中暑暴血 + 出血（§5）
    # 失温：体温条触底
    if temperature <= 0:
        if randi() % 50 > 28:                   # 22/50 概率 ✅ §7
            is_sick = true                      # 起感染（原 iv[34]=1）
```

- **中暑 Heatstroke**：即时体温 > 100 → 生命暴跌至 5 + 出血（原版 §5）。本项目热源叠加极多时才可能触发，与原版一致。
- **失温 Hypothermia**：体温条 ≤ 0 → 22% 起感染。原版还有每秒 `iv[15]-=0.25` 的冷伤——本项目 `temperature==0` 已通过现有 `update_status()` 的 `BAD_STATUS_HEALTH_DECREASE`（0.3）扣血（`int(temperature==0)` 那一项），**等价已覆盖**，无需另写冷伤。

### 5.6 感染（Infection）接入现有 `is_sick`

`PlayerStatus` 已有 `is_sick`（驱动 `sick_state_changed`）、`is_health()`（要求不 sick 才回血）。补两件事：

1. **感染附加掉值**：`is_sick` 时每 5 秒额外 `hunger -= 1; thirst -= 1`（§2/§7）。在 `update_status()` 内用一个 5 秒计数器（`survival tick % 5`）触发，或简化为每秒 `-0.2`（等效摊薄）。
2. **自愈**：原版 `hunger≥80 且 thirst≥80 且 temperature≥80 且不出血` 时 50% 自愈。现有 `is_good_status()`（≥75 三项）已近似——`update_status()` 末尾 `if is_sick and is_good_status(): is_sick = false` **已存在**，可保留或收紧到 ≥80 + 50% 概率以更贴原版（见 §7 待定）。

> `is_health()` 回血门控（`temperature>=50` 等）已就位，体温一旦掉破 50 自动停回血——无需改动，**只要让 temperature 真的会变**即可。

## 7. 决策

### 已定 ✅
1. **夜幕用 `CanvasModulate`**（非全屏 ColorRect）：一节点全局、不盖 UI、Tween `color` 平滑。`night_overlay` 贴图仅作可选氛围叠层。
2. **体温条方向**：100=暖 / 0=冷（与原版 `iv[16]` 一致，`is_health()` 已按此设 `temperature>=50`）。
3. **不另起体温计时器**：结算并入现有 `PlayerStatus.update_status()` 1 秒 tick，原版 5 秒增量按 1/5 摊到每秒。
4. **热源契约用 Area2D + 分组 `"heat_source"`**（方案 A）：10/11 加组并设 `heat_value`/`is_eave` 即接入，不改 PlayerStatus。
5. **公式逐项忠实**：日寒被三件衣物保暖之和抵消；夜/雨另扣；篝火+15、地堡+5、载具+4、暖区+heat、snowborn+2、sweethome 暖区翻半。
6. **失温冷伤复用现有逻辑**：`temperature==0` 已被 `update_status()` 的 `BAD_STATUS_HEALTH_DECREASE` 扣血覆盖，不重复实现。
