## 0. 两条贯穿全局的设计原则（所有分文档遵守）

1. **Godot 惯用实现**：用节点/场景继承、信号、`Resource`(`.tres`) 数据、`autoload` 单例、`Timer`、`Area2D`/物理层、`AnimationTree`、`NavigationServer` 等引擎原生机制表达逻辑——**不照搬** Construct2 的整数 `iv[]`、魔法数字 code、按距离 create/erase 的平台妥协。
2. **原版功能与数值忠实复刻**：行为、手感、关键常量（伤害/速度/衰减率/冷却/概率/容量）**对齐原版**。代码结构可 Godot 化，但**玩家可感知的结果必须一致**。每个数值在分文档中标注原版出处。

> 一句话：**「壳用 Godot，芯用原版数值」**。

## 2. 对现有文档的两处事实纠正（实测代码后）

> 现有 `02`/`04` 写于库存接口未明朗时，有两处**过时假设**，规划时以本节为准：

1. **「背包/拾取 API 尚未接通」——已过时。** `character/player/inventory.gd:37 put_to_inventory(item, from_world)` 已**完整实现并接线**（`detection_area.gd` 拾取、`item_action_table.gd` 合成/换弹都在用）。**给玩家发物品的标准链路现已可用**：
   ```gdscript
   var item := ItemCreator.create_item_from_id(food_id)
   player.inventory.put_to_inventory(item, false)   # from_world=false：物品不在世界树里
   ```
   → 因此 `04` 植物采集、`pump_interaction`、`searchable_interaction`、`tree.chop` 的「发物品」TODO **不再被库存阻塞**，只差①调上面两行 + ②`05` 的 loot 池决定「发什么」。

2. **体温（temperature）有字段、无逻辑。** `status/player_status.gd` 里 temperature 是个 0–100 字段并参与健康判定，但**没有任何衰减/回升/外部计算**——原版那套「夜冷/雨冷/昼热 + 服装保暖 + 篝火/庇护所/车厢 + 天赋」公式整体缺失。归入 `08` 实现。

## 3. 依赖图（谁阻塞谁）

```
物品/库存(✅) ──┬─→ 05 战利品刷新 ──┬─→ 02/03 容器·车有东西可搜
               │                   └─→ 04 采集发物品、tree 掉木材
               ├─→ 06 战斗 ──→ 07 僵尸(要能被打死) ──→ 09 经验(击杀给 XP)
               └─→ 08 消耗品改状态(已通)

01 地图生成 ──→ 03 地块模板 ──→ 城镇长出来 ──→ 07 僵尸刷新点、05 loot 点有载体
                                            └─→ 11 载具/加油站落位

08 体温 ←──┬── 昼夜(提供昼/夜/时段)
           ├── 天气(提供雨/暴雪)
           ├── 服装(✅ gear.heat 已有字段)
           └── 10 建造(篝火/帐篷提供热源) + 11 载具(车厢保暖)
```

## 4. 推荐执行顺序（两条并行线）

**A 线 · 世界生成**（已在推进，延续 `01`→`03`）：
```
01 地图数据(✅) → 02 环境物体(🟡) → 03 地块模板 → 城镇成形 → 01 收尾(小地图/直升机/地堡)
```

**B 线 · 玩法循环**（本轮规划重点，让游戏「能玩」）：
```
05 战利品刷新 → 06 战斗伤害 → 07 僵尸AI(+动物) → 09 经验天赋
                                              → 08 体温昼夜天气(生存压力)
```

**会合后（扩展）**：`04 采集` → `10 建造种植` → `11 载具` → `12 存档/模式`。

> 里程碑建议：
> - **M1「能搜刮」**：05 + 02/03 容器 → 进城能开箱拿枪。
> - **M2「能战斗」**：06 + 07 → 僵尸会追会打、玩家能反杀。
> - **M3「能生存」**：08 + 09 → 昼夜/体温/天气施压，击杀升级解锁天赋。
> - **M4「能扩展」**：04/10/11 → 种田、建营地、开车。
> - **M5「能留存」**：12 → 存档 + 游戏模式 + 角色解锁。

## 附录 A · 物品/库存系统实测结论（无需单独文档）

> 为何不给「物品/库存」立设计文档：实测已近乎完整，列此备查。

**已完整（100%）**：物品类层级（`Item`→`SlotItem`/`DurabilityItem`→各类）；全 `.tres` 资源（食物/药/弹/护具/近战/枪）；库存 9 装备槽 + 护具子槽 + 堆叠 + 按耐久排序 + 拖拽/交换/丢弃/拆分；拾取（自动近身 + 手动 `interact` + 寻路接近）；消耗品改状态；换弹/退弹；合成（9 硬编码配方 + 拖拽触发）；全套库存/状态 UI。

**待补小尾巴**（并入相邻系统顺手做）：
- `item/item_creator.gd:9` `create_item_from_id()` 缺 `range(190,202)` 手枪段 → 程序化造手枪会断言失败（并入 `05`/`06`，发枪 loot 时会用到）。
- `character/player/inventory.gd:106` `find_item()` 返回 null 桩 → 合成/换弹按 ID 找物时需要（并入 `06`）。
- 耐久磨损：`DurabilityItemResource.wear_per_use` 有字段，无「使用即磨损」逻辑 → 并入 `06`（武器开火/挥击）、`08`（服装受击掉耐久减保暖）。

**核心 API（其它系统调用入口）**：
- `PlayerInventory.put_to_inventory(item: Item, from_world := true) -> bool`
- `ItemCreator.create_item_from_id(id: int) -> Item` / `create_random(script)`
- 信号 `equipment_changed(type, item)` / `slot_item_changed(type, item)`
