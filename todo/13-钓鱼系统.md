# 钓鱼系统设计（鱼竿 + 鱼窝 + 抓取掷骰）

> 资源：`asset/images/item/basic_fishing_rod.png`、`spinning_fishing_rod.png`、`asset/images/item/{herring,salmon,ruffe,perch,small_fish_fillet,big_fish_fillet}.png`、原版鱼窝贴图 `images/fish_nest-sheet0.png`（对象 `t833`，5 帧动画）。
> 定位：钓鱼是「**装备鱼竿 → 在鱼窝旁使用鱼竿 → 等 Timer → 掷骰抓鱼 → 入背包**」的一条交互/状态流。鱼竿物品与鱼物品**都已存在**（`item/craft.gd`、`item/number_item.gd`），本文只补**鱼窝场景 + 抓取流程 + 战利品表 Resource**。鱼窝是一类环境物件（参照 `02-环境物体场景.md` 的「交互点」一档，与水泵同列），落在水域边缘。
> 调研来源：`origin/others/文档/datajs_export/game_events.decoded.txt`（行 19383–19446 钓鱼掷骰逻辑 / 行 32846–32918 鱼竿合成）、`物品类型与数值.md`（§3.2 行 104–109 鱼营养值）、`数值探索/合成与制作配方.md`（行 42 鱼竿配方）、`datajs_export/objects/object_833.json`（鱼窝对象）、`datajs_export/parsed/item_names.json`（行 82–96 原版 ID 名）。

## 0. 原版机制还原（必须先厘清）

原版钓鱼**不是钓鱼小游戏**，而是**纯计时 + 一次掷骰**。核心逻辑见 `game_events.decoded.txt` 行 19383–19446，函数 `item_activate`（对鱼竿物品执行 use 动作时进入）：

```text
- IF CompareParam(0, =, 86)            # 使用的物品 = 旋转鱼竿（Spinning, 原版 ID 86）
  - IF IsBoolInstanceVarSet(iv[44])    # 玩家「忙碌」标志 → 拒绝，提示报错
    > 报错 client_log
  - IF Else()
    - IF t172.IsOverlapping(t833)       # 玩家与「鱼窝」对象 t833 重叠
    - IF PickRandom(t833)
      - IF t833.iv[0] 置位               # 鱼窝类型 A（梅花鲈塘）
        > 鱼竿耐久 -10                    # Param(1) - 10
        > iv[21] = 1                      # 玩家锁定（播放钓鱼动作）
        > t833.Destroy()                  # 鱼窝消耗销毁
        > Wait(4)                         # 等待 4 秒
        > iv[21] = 0                      # 解锁
        - IF choose(1,2,3,4,5) == 1      # 1/5 概率空军
          > "钓鱼失败" 红字
        - IF Else()                       # 4/5 概率上钩
          > Check_space_inventory(-1, choose(93,93,93,93,94))  # 75% 梅花鲈 / 25% 河鲈
          > "钓到鱼" 绿字
      - IF t833.iv[1] 置位               # 鱼窝类型 B（鲱鱼塘）
        > ……同上，产出 choose(91,91,91,91,92)               # 75% 鲱鱼 / 25% 鲑鱼
    - IF Else()                           # 没站在鱼窝旁
      > "这里没有鱼" 红字

- IF CompareParam(0, =, 87)            # 基础鱼竿（Basic, 原版 ID 87）
  > 同上，唯一区别：耐久 -25（Param(1) - 25）
```

**两根鱼竿的抓鱼概率表完全一致，唯一差别是耐久消耗**（旋转 -10 / 基础 -25，即旋转竿更耐用）。原版**未发现**两竿在鱼种质量上的差异。

### 几个还原关键点

| 项 | 原版行为 | 出处 |
|---|---|---|
| 触发方式 | **对鱼竿物品执行 use 动作**（`item_activate`），不是点水面交互 | `game_events` 19383/19415 `CompareParam(0,=,86/87)` |
| 地点条件 | 玩家须与**鱼窝对象 `t833`** 重叠（`IsOverlapping(t833)`），而非站在任意水块旁 | 19388/19420 |
| 鱼窝两型 | `t833.iv[0]`=梅花鲈塘 / `t833.iv[1]`=鲱鱼塘，决定产出鱼种池 | 19389–19410 |
| 等待时长 | `Wait(4)` = **4 秒**（期间 `iv[21]=1` 锁玩家） | 19394/19405/19426/19437 |
| 空军率 | `choose(1,2,3,4,5)==1` = **1/5（20%）** 钓不到 | 19396/19407/19428/19439 |
| 成功产出 | 80% 命中里：**75% 普通鱼 / 25% 高级鱼**（`choose(x,x,x,x,y)`） | 19399/19410/19431/19442 |
| 鱼窝消耗 | 每次抓鱼**销毁该鱼窝 `t833`**（一次性，非反复钓同一窝） | 19392/19424 `t833.Destroy()` |
| 耐久消耗 | 旋转竿 **-10/次**、基础竿 **-25/次** | 19391/19423 |
| 满耐久 | 鱼竿出生耐久 100（合成）或 `choose(50,75,100)`（刷出） | 32917 / 47519 |

> ⚠️ **重要差异**：原版鱼窝 `t833` 抓一次即销毁——这意味着「鱼窝」是**有限的可消耗资源点**，不是反复钓的固定钓位。本项目可二选一（见 §7 决策 D-鱼窝消耗）：① 忠实销毁（鱼窝需重生 Timer，类似浆果灌木）；② 改为冷却复用（更友好）。本文设计默认走 **①销毁 + 重生 Timer**，以贴合原版。

## 1. 数值 / 规格

### 1.1 抓鱼掷骰（两竿共用）

```text
使用鱼竿（玩家与鱼窝重叠）：
  ├─ 玩家锁定 4s（is_busy 期间不可移动/再操作）
  ├─ 鱼窝销毁，鱼竿耐久 -wear_per_use
  └─ 4s 后掷骰：
        20% → 空军（提示「钓鱼失败」）
        80% → 上钩，按鱼窝类型取池：
              梅花鲈塘：75% 梅花鲈(Ruffe) / 25% 河鲈(Perch)
              鲱鱼塘  ：75% 鲱鱼(Herring) / 25% 鲑鱼(Salmon)
```

## 2. 触发模型（与现有代码对接）

原版是「对鱼竿 use」。本项目两种落地方案：

| 方案 | 描述 | 优 | 劣 |
|---|---|---|---|
| **A. 物品动作（贴原版）** | 在 `ItemAction` 里给鱼竿加一个 `'Fish'` 选项（仿 `item_action_table.gd` 的 `'Consume'`/`'Eject'`），执行时检测玩家是否在鱼窝旁 | 完全还原；不必装备 | 需让 item_action 拿到「玩家/鱼窝重叠」上下文，物品动作目前是纯背包操作 |
| **B. 水边交互（贴 Godot 习惯）** | 鱼窝场景挂 `InteractionArea`（仿水泵/砍树），玩家进范围按 `interact`；交互时检查**装备槽是否有鱼竿** | 复用现成 `InteractionArea` + `interact` 输入，与门/水泵/砍树同范式；上下文天然（交互发起方即鱼窝） | 略偏离原版「use 鱼竿」语义（改成「对鱼窝交互」） |

**本文采用方案 B（鱼窝挂 `InteractionArea`）**，理由：与 `02/04` 既定的交互范式统一（`chop_interaction.gd`/`pump_interaction` 都是「对物件交互」），且鱼窝重叠检测在原版本就是前提，把鱼窝做成交互体最自然。鱼竿则作为**前置条件**（背包/装备里有鱼竿才可交互），等价于原版「拿着鱼竿才能钓」。
> 决策见 §7 D-触发。若后续要更贴原版，可在方案 B 基础上**追加**方案 A 的物品动作入口，二者复用同一段抓鱼逻辑。

### 2.1 水域 / 鱼窝邻近检测

原版用「与鱼窝对象重叠」而非「靠近水块」。本项目鱼窝是放在**水域边缘**的环境物件，因此：

- **首选**：鱼窝场景自身**就是交互体**（`InteractionArea` + `CollisionShape2D`），玩家进范围即可钓 → **无需额外查水块**。鱼窝在地图生成时由水域装饰流程放到水边（见 §4）。
- **可选校验**：抓鱼前可顺手查一次脚下/目标点是否真为水 tile，避免鱼窝被错放到陆地。查询走 `ground.gd` 现成接口：
  - 块级：`ground.is_water(ground.get_block(bx, by))`（`environment/ground.gd` 行 242–244）。
  - tile 级：`var c := water_layer.local_to_map(water_layer.to_local(point)); water_layer.get_cell_source_id(c) != -1`（`WaterLayer` 是 `ground.gd` 的 `$WaterLayer`，行 4）。
  - > 查 `class_tilemaplayer.md` 确认 `local_to_map()` / `get_cell_source_id()` 签名（返回 -1 表示空格）。

## 3. 节点结构与状态机

### 3.1 鱼窝场景 `fish_nest.tscn`（沿用交互物件范式）

参照 `04-植物采集.md` 的 `HarvestPlant` 与 `chop_interaction.gd`：鱼窝 = 一个可交互、抓一次即「枯竭」、按 Timer 重生的水边物件。

```text
FishNest (Node2D)                    @tool, class_name FishNest，持有掷骰逻辑
├── Sprite (Sprite2D)                fish_nest-sheet0（原版 5 帧；活跃帧 / 枯竭帧）
├── InteractionArea (实例 mechanics/interaction_area.tscn)
│   └── CollisionShape2D             玩家进范围才可钓
│       └─ FishInteraction.gd        extends InteractionArea, 重写 _on_interact()
├── Timer                            枯竭后重生倒计时（一次性）
└── [可选] CastSound (AudioStreamPlayer2D)
（无 StaticBody：水边可踩近，鱼窝不挡路）
```

- `@export var nest_type: NestType`（枚举 `RUFFE_POND` / `HERRING_POND`）→ 对应原版 `t833.iv[0]/iv[1]`，决定鱼种池。
- `@export var catch_table: FishCatchTable`（`.tres`，见 §3.3）。
- `@export var respawn_time: float`（鱼窝重生冷却，原版未给数值 → **待核实**，暂用浆果灌木的 `LOOT_RESPAWN_TIME ≈ 1480s` 同量级，Inspector 可调）。
- 复用 `InteractionArea` 现成成员：`enabled` / `one_shot` / `interact()` / `_on_interact()`（`mechanics/interaction_area.gd`）。

### 3.2 抓鱼状态机（挂在 `FishNest` / `FishInteraction`）

纯计时，无 bite-window（贴原版）：

```text
IDLE（鱼窝活跃，enabled=true）
  │  玩家交互(_on_interact) + 持有鱼竿 + 鱼窝可用
  ▼
CASTING（锁玩家 is_busy，鱼竿 wear()，鱼窝 enabled=false）
  │  Timer(cast_wait_time = 4.0) 倒计时
  ▼
RESOLVE（4s 到点，掷骰）
  ├─ 20% FAIL → 提示「钓鱼失败」
  └─ 80% SUCCESS → catch_table.roll() 选鱼 id → 入背包 → 提示「钓到鱼」
  │  （解除 is_busy）
  ▼
DEPLETED（鱼窝枯竭：销毁 或 启动 respawn Timer）
  │  Timer.timeout
  ▼
IDLE（enabled=true，可再钓）
```

> 玩家锁定（原版 `iv[21]`）：用玩家已有的「忙碌/动作锁」机制（若 `player` 暂无此字段，先以 `await get_tree().create_timer(4.0).timeout` 串起流程，玩家锁后续接入，**与现有 pump/searchable 的 TODO 占位风格一致**）。

### 3.3 战利品表 Resource `FishCatchTable`（`.tres`）

把「鱼窝类型 → 加权鱼 id 池 + 空军率」做成 `Resource`，仿 `02` 的 `LootPoint`/`04` 的产出表思路，避免硬编码 `choose(...)`：

```gdscript
@tool
class_name FishCatchTable extends Resource

## 空军率（原版 1/5）
@export var miss_chance: float = 0.20

## 加权鱼种：id → 权重（原版 choose(93,93,93,93,94) = {74:3, 73:1}，本项目鱼 id）
@export var weighted_fish: Dictionary = {}

## 命中后按权重选一个鱼 id；-1 表示空军
func roll() -> int:
    if randf() < miss_chance:
        return -1
    var total := 0.0
    for w in weighted_fish.values():
        total += w
    var r := randf() * total
    for id in weighted_fish:
        r -= weighted_fish[id]
        if r <= 0.0:
            return id
    return weighted_fish.keys().front()
```

预置两份 `.tres`（权重比 3:1 = 原版 75/25）：

| 文件 | `miss_chance` | `weighted_fish`（本项目 id） | 含义 |
|---|---|---|---|
| `mechanics/res/fish_table_ruffe_pond.tres` | 0.20 | `{74: 3.0, 73: 1.0}` | 75% 梅花鲈(74) / 25% 河鲈(73) |
| `mechanics/res/fish_table_herring_pond.tres` | 0.20 | `{72: 3.0, 75: 1.0}` | 75% 鲱鱼(72) / 25% 鲑鱼(75) |

> 两份表的 `miss_chance` 与权重比都来自原版掷骰，**已核实**；具体鱼 id 用本项目重编号（72/73/74/75）。

### 3.4 交互脚本 `FishInteraction.gd`（仿 `chop_interaction.gd`）

```gdscript
class_name FishInteraction
extends InteractionArea


func _on_interact() -> void:
    var nest := owner as FishNest
    if not nest:
        return
    nest.try_fish()        # 持竿校验 + 掷骰 + 入背包 + 枯竭，全部在 FishNest 内
```

`FishNest.try_fish()` 要点（伪流程，结合 §3.2）：
1. 取玩家装备/背包里的鱼竿（`player.inventory` 查 `Craft` id∈{93,94}）；无竿 → 提示「需要鱼竿」并 return。
2. `enabled = false`（防重复触发）；`rod.wear()`（`DurabilityItem` 自带磨损，耐久归零自动销毁）。
3. `await get_tree().create_timer(4.0).timeout`（cast_wait_time）。
4. `var id := catch_table.roll()`；若 `id < 0` → 空军提示；否则：
   `var it := ItemCreator.create_item_from_id(id); player.inventory.put_to_inventory(it, false)`（`item/item_creator.gd` + `character/player/inventory.gd` 行 37）。
5. 进入 DEPLETED：销毁 或 `respawn_timer.start(respawn_time)`（见 §7 D-鱼窝消耗）。

> 「给玩家加鱼」这一步与 `04-植物采集.md §5` 同样依赖背包接通。**集成事实已确认**：`ItemCreator.create_item_from_id(72..75)`（落在 0–85 → `NumberItem`）+ `inventory.put_to_inventory(it, false)` 可直接发鱼。若钓鱼先于背包系统验收，可像现有 pump/searchable 一样先留 `TODO` 占位。

## 4. 世界生成位置

- 鱼窝是**水域边缘装饰物**。`ground.gd` 已有水域块（`BlockType.WATER`，行 412–422）+ 水边装饰散布（`_scatter_water_block` / `water_edge_scattering`，行 665–666）。
- 鱼窝按「水边」散布：在 `WATER` 块与陆地交界的格上低概率放 `fish_nest.tscn`（参照 `04` 野生作物的「森林模板程序化散布」，但锚到水缘）。
- 鱼窝两型（`RUFFE_POND`/`HERRING_POND`）放进对应 `TemplateSlot.choices` 或水缘散布池随机二选一（对应原版 `t833.iv[0]/iv[1]` 随机置位）。
- 交叉引用：水块识别/坐标换算见 `01-地图生成.md`（`WaterLayer`、`generate_water_blocks`）；环境物件「交互点」分类见 `02-环境物体场景.md §1`（鱼窝与水泵同列）。
