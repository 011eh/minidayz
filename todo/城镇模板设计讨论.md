## 一、先澄清：「槽位」不是建筑专属

你问「除了建筑是否还有其他的需要槽位」——**是的，槽位是通用容器，不止建筑**。原版 `t518` 每行就是一个槽位，靠 `kind` 区分往里放什么。城镇模板里这 5 类 `kind` **全都用固定槽位**：

| kind       | 放什么         | 村庄9号模板里的体现  | code 规则               |
| ---------- | -------------- | -------------------- | ----------------------- |
| **7 建筑** | 房子/车库/超市 | 11 个槽              | `choose(9..13)` 随机色  |
| **6 车辆** | 17 种车        | 3 个槽               | `random(12)` 随机车型   |
| **1 树**   | 松/阔叶/灌木   | **9 棵固定位置树**   | `choose(1..6)` 随机树种 |
| **3 特殊** | 信箱/牌子      | 1 个信箱             | `2` 固定                |
| **4 容器** | 桶/箱          | (村庄无，别的模板有) | 固定                    |

**唯一不用槽位的是「野外森林」**（模板 1/2/8/10）：它是 `Repeat(16)` 在 4×4 子格程序化散树，位置纯随机。所以要分两套机制：**城镇 = 固定槽位模板**，**野外 = 程序散布**（你 todo 里 C5 已单列，正确）。

还有一个细节：**有的槽位连位置都带随机**——村庄的车辆槽 `dx = +100 + random(300)`，所以「槽位」要允许「位置抖动范围」，不只是死坐标。

## 二、完整零件清单（原版 → Godot 映射）

要把这个功能**实现 + 应用**，需要下面 8 个零件。标 ⭐ 的是 MVP（先做村庄就能看到成果），标 ○ 的可延后：

| #   | 零件                   | 干什么                      | 原版                                    | Godot 实现                                                                | MVP    |
| --- | ---------------------- | --------------------------- | --------------------------------------- | ------------------------------------------------------------------------- | ------ |
| 1   | **槽位元素**           | 描述「一个位置放什么」      | `t518` 行 `[kind,code,dx,dy]`           | `TownSlot` 节点（`Marker2D`+导出字段），**位置=节点自身坐标，编辑器可拖** | ⭐     |
| 2   | **模板本体**           | 一套城镇的蓝图（一堆槽位）  | `generate_location` 的 `iv[0]` 分支     | `template_village_9.tscn`（`Node2D` 根 + 一堆 `TownSlot` 子节点）         | ⭐     |
| 3   | **类型→模板集 注册**   | 村庄随机选 9/4/16/17 哪套   | step5 `choose(9,4,16,17)`               | 字典 `BlockType → Array[PackedScene]`，`choose` 一套                      | ⭐     |
| 4   | **code→资源 映射**     | 编号 9 → village_brown.tscn | §7 建筑表                               | 字典 `code→PackedScene`；**或槽位直接存 `PackedScene` 免掉此表**（见下）  | ⭐     |
| 5   | **应用器**             | 遍历地点→选模板→展开→实例化 | `create_map_element` kind=7             | `ground.gd` 里一个循环，`instantiate()` 到 `hotspot+slot位置`             | ⭐     |
| 6   | **随机助手**           | `choose` / 位置抖动         | `choose()`/`random()`                   | 复用现有 `rng`，从 `Array` 随机取 + `randf_range` 抖动                    | ⭐     |
| 7   | **野外散树器**         | 空地程序化森林              | `Repeat(16)` 4×4 子格                   | 独立函数，非模板（按 hotspot 撒树场景）                                   | ○      |
| 8   | **建筑场景自包含内部** | 楼里要有墙/碰撞/僵尸点/物资 | kind=7 还 spawn 墙/阴影/僵尸刷新点/loot | **由各 `Building.tscn` 自带**，模板层不管                                 | 依赖项 |

> **第 8 项是隐藏依赖**：模板只负责「把哪栋楼放哪」，楼内部的碰撞、内饰、僵尸刷新点、可搜刮物资必须由建筑场景自己包含。否则城镇生成出来是「空壳房子」。需要确认你现有的 `Building` 基类/场景已经（或将要）自带这些——这决定了「城镇模板」这条线到哪算「完成」。

## 三、推荐方案：槽位即节点、模板即场景（贴合你的诉求）

你想「在编辑器里调槽位位置」——那最顺的就是**让槽位本身是个节点**，位置即节点坐标，所见即所得。

**零件1 `TownSlot`**（`@tool`，挂在 `Marker2D` 上）：
```gdscript
@tool
class_name TownSlot extends Marker2D
enum Kind { BUILDING, VEHICLE, TREE, PROP, CONTAINER }
@export var kind: Kind = Kind.BUILDING
@export var choices: Array[PackedScene] = []     # 随机集；填 1 个=固定
@export var position_jitter: Vector2 = Vector2.ZERO   # 位置抖动范围(村庄车辆用)
# 位置：直接用节点的 position —— 编辑器里拖拽即所得 ✓
func _draw():  # @tool 下画个框/图标，摆位时看得见
```

**零件2 模板场景** `template_village_9.tscn`：一个 `Node2D` 根，下面摆 11+ 个 `TownSlot`。你在编辑器里直接拖动每个 slot 到位，`choices` 里拖入候选建筑场景。

**零件5 应用器**（`ground.gd`）:
```gdscript
for loc in locations:
    var tpl: PackedScene = choose(TEMPLATES[loc.type])  # 零件3
    var town := tpl.instantiate()
    town.position = loc.hotspot
    add_child(town)
    for slot in town.find_children("*", "TownSlot"):
        var scene: PackedScene = choose(slot.choices)   # 零件6 槽内随机
        var obj := scene.instantiate()
        obj.position = slot.position + random_jitter(slot.position_jitter)
        town.add_child(obj)
        slot.queue_free()   # 槽位是占位标记，用完删
```

**这个方案的一个红利**：如果 `TownSlot.choices` 直接存 `PackedScene`，**零件4（code→资源表）就消失了**——原版用整数 code 是 Construct2 所迫，Godot 里直接引用场景即可，少一层映射、少一处易错。代价是「全局换某栋楼」要改多个 slot（但「换色」本来就是逐槽的事）。

**每个城镇=一个子树节点**，还顺带满足你阶段 D3 想要的「按 town 粒度做可见性门控」——直接对这个 town 节点挂 `VisibleOnScreenEnabler2D` 即可。

## 四、三种存储方案对比（你可挑）

| 方案                                         | 摆位方式                   | 优点                                   | 缺点                       |
| -------------------------------------------- | -------------------------- | -------------------------------------- | -------------------------- |
| **A. 场景+TownSlot节点**（你倾向，我也推荐） | 编辑器画布拖拽，所见即所得 | 直观、每town是子树好管理、免code表     | 模板多时场景文件多         |
| B. GDScript const 槽位表                     | 写死坐标数字               | 数据集中、改批量快、可直接抄原版 dx/dy | 摆位靠脑补坐标，不可视化   |
| C. 自定义 Resource(.tres)                    | Inspector 填数             | 结构化、可校验                         | 坐标仍是数字，非画布可视化 |

A 和 B 不互斥——可以**先用原版 dx/dy 数据初始化生成一个模板场景**（脚本一次性铺 TownSlot），之后你在编辑器里微调。这样既省了手摆 11 个槽的力气，又保留可视化调整能力。
