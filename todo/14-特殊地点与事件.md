> **架构总原则**（沿用 `01` 的决定）：原版「按距离 create/erase」是 HTML5 所迫，本项目不采用。残骸/入口在世界生成时**一次性实例化**进场景树，屏幕外由引擎剔除 + 可见性门控（见 `01` 阶段 D）。空投箱是运行期事件物，地堡是独立场景。

## 尺度规格（沿用 `01`）

- 地表世界：16×16 block，格距 **1020px**，地点世界坐标 `(700 + gx*1020, 320 + gy*1020)`，居中偏移 `+500`。残骸/入口都挂在某个地点标记上 → 落点 = `地点世界坐标 + (500, 500)` 一类的固定偏移（见各节）。
- 地堡世界：原版放在 **远离地表的另一片坐标**（`CreateObject(t779, "bg", 2306, 17464)`，即 Y≈17464，整张地图之外）。本项目改为**独立 `.tscn` 场景**，不必塞进同一坐标系（见 §2 决策）。

## 1. 直升机 / 悍马残骸（步骤 8 `generate_helicrashes`）

### 1.1 生成规则（✅ `map_generation.decoded.txt:7094-7143`）

世界生成管线步骤 8。按 `CurrentLevel` 决定两种残骸的数量：

| CurrentLevel | helispawned（直升机残骸） | hammerspawned（悍马残骸） |
|:---:|:---:|:---:|
| 0 | **0** | **1** |
| 1 | **2** | **3** |
| 2 | **3** | **5** |
| 3 | **4** | **6** |

> 即关卡 0 没有直升机残骸，只有 1 个悍马残骸；关卡越深残骸越多。

**落点规则**（两个 `While` 循环，逐个递减计数到 0）：

- **直升机残骸（t335）**：随机挑一个 `iv[9]==0`（未占用）的**野外/森林/秘密格**——模板号 `iv[0] ∈ {1, 10, 2, 8}`（森林/野外/秘密）。挑中后置 `iv[9]=1` 占用，在 `(地点X+500, 地点Y+500)` 吐出残骸，记 `iv[1]=所属地点UID`。
- **悍马残骸（t427）**：随机挑 `iv[9]==0` 的**军营(5)/消防站(15)/道路(13)格**。挑中置 `iv[9]=1`，在 `(地点X+500, 地点Y+600)` 吐残骸，记 `iv[2]=所属地点UID`。
- **一次性、不重生**：占用标记 `iv[9]` 确保同格不重复；游戏期间不再刷新（**one-shot**）。

> ⚠️ 与地堡入口（步骤 9）**共用 `iv[9]` 占用位且抢同一批森林格**：步骤 8 先跑、占掉一些森林格，步骤 9 再在剩余森林格放地堡入口（避免重叠）。本项目须保持这个**先残骸后地堡**的顺序。

### 1.2 残骸内容（✅ `game_events.decoded.txt:3679-3695`）

直升机残骸走近时（原版惰性展开，本项目直接在场景里）：

- 吐出残骸主体 `t237` + 大碰撞体（`t339` AnimFrame 19）。
- 布置 **3 个 loot 点**（`t209`，`iv[0]=15`）——`15` 是该 loot 池编号，由战利品刷新系统统一刷（**→ 05-战利品刷新系统**拥有 21 池映射）。
- 同时生成一个**僵尸刷新区**（`t398` 300×300 + `Trigger_spawn`）——残骸是高危高回报点（**→ 07-僵尸系统**拥有刷新逻辑）。
- **悍马残骸**：主体 + 碰撞，loot 由其各自分支决定（本项目按 `02 §1` 已有 `hammer_crash_h/v` 残骸场景，挂 `LootPoint` 锚点即可，池号 待核实）。

### 1.3 本项目场景

```
HeliWreck.tscn  (Node2D, y_sort 子节点)
├── Sprite        Sprite2D            (helicrash 残骸贴图, centered=false, 脚底对齐原点)
├── Body          StaticBody2D        (layer 2 世界实体)
│   └── Shape      CollisionShape2D    (残骸占地大矩形)
├── LootPoints     Node2D
│   ├── LootPoint1  Marker2D (loot_type = 15)   # → 05 刷新
│   ├── LootPoint2  Marker2D (loot_type = 15)
│   └── LootPoint3  Marker2D (loot_type = 15)
└── ZedSpawn       Marker2D            # → 07 在此挂刷新区(半径~150)
```

`HammerWreck.tscn` 同构，少一组 loot/zed（按原版差异，悍马 loot 较少，**待核实** 池号）。已有 `hammer_crash_h/v.tscn`（`02 §1` P0-2）可直接派生为本场景。

### 1.4 生成接入

世界生成（`environment/ground.gd`，`01` 阶段 A 已写好 `grid` 与各地点 hotspot）跑完地点/道路后，由 `EventSystem` 或 ground 的收尾步骤执行 `_place_wrecks()`：

```gdscript
# 伪代码：按关卡数量，从候选 hotspot 随机挑、标记占用、实例化
func _place_wrecks(level: int) -> void:
    var heli := HELI_COUNT[level]      # [0,2,3,4]
    var hammer := HAMMER_COUNT[level]  # [1,3,5,6]
    _spawn_n(heli,   _candidates(BlockType.FOREST | SECRET), HeliWreck,   Vector2(500,500))
    _spawn_n(hammer, _candidates(BlockType.MILITARY | FIRESTATION | ROAD_H), HammerWreck, Vector2(500,600))
```

`_candidates()` 复用 `grid` + 各 block 的 hotspot（`01 A5`）；占用集合替代原版 `iv[9]`。

## 2. 地堡（步骤 9 入口 + `Bunker_generate` 地下副本）

地堡分两半：**地表入口**（步骤 9 落子）与**地下世界**（进入时展开的独立场景）。

### 2.1 地表入口生成（✅ `map_generation.decoded.txt:7144-7153`）

- **门槛**：`CurrentLevel ≥ 1` 才生成（关卡 0 无地堡）。
- **落点**：随机挑一个 `iv[9]==0` 的**森林/野外/秘密格**（`iv[0] ∈ {1,10,2,8}`，与直升机残骸同候选池，靠 `iv[9]` 占用避让），在 `(地点X+500, 地点Y+500)` 吐出入口 `t780`。
- **数量**：原版每图 **1 个**（无计数循环，单次 `PickRandom` 落一个）。

### 2.2 进入：展开地下副本（✅ `game_events.decoded.txt:13156-13270`）

玩家碰到入口 `t780`（且非建造模式、玩家未在载具）触发 `Level_menu` 确认菜单 → 调用 `Bunker_generate(深度参数)`：

1. 创建地堡 tilemap `t779` + 碰撞 `t781`（地面 + 墙）。
2. 把玩家 `t184` 瞬移到地堡入口坐标（`SetPos(8425, 18450)`）。
3. 置 `inBunker = 1`；强制白天（`Day_mode=1`，关夜晚层与雨）；停雨声/昼夜环境音；淡入过渡（`t663.StartFade`）。
4. 卸载地表地点实体（`erase_map_location`，省内存）；重跑 `Zed_check`。
5. 按**深度参数**（`Bunker_generate` 的 Param0）布置内部：门 `t782`、装饰箱 `t792`、可搜刮箱 `t793`（`iv[0]=51/52` loot 池）、僵尸刷新点 `t791`、出口 `t790`。
   - Param0=1 与 Param0=2 是**两套不同的内部布局/loot 密度**（深层更多/更好）。
6. 出口 `t790`：玩家碰到 → `Level_menu(12)` 确认 → 回地表（清地堡、`inBunker=0`、玩家瞬移回入口附近）。

> **深度/层级机制（待核实）**：原版 `Bunker_generate` 至少有 Param0 ∈ {1,2} 两套布局；箱内 `iv[0]=51/52` 是两档 loot 池。是否有更深层（递归下探）、深度如何决定，截取未完整 → 标 **待核实**，先实现 1~2 层。
> **深度 → 被动经验（→ 09）**：`12`/`09` 提到地堡深度关联**被动 XP 速率**——越深 XP 越快、风险越高。具体倍率 **待核实**，接口预留在 `BunkerWorld.depth`，由 `09` 的被动 XP 系统读取。

### 2.3 本项目：地堡是**独立场景**（决策）

原版把地堡塞进同一世界坐标系外的远处（Y≈17464）+ `erase_map_location` 卸地表，是 Construct2 单 layout 的妥协。**本项目用 Godot 场景切换**——地堡是一个独立 `BunkerWorld.tscn`，进入 = `change_scene` 或 push 一个子场景；退出 = 返回地表场景并恢复玩家状态。这样：地表实体无需卸载（保留在挂起的地表场景）、白天/雨切换是地堡场景自带的环境、玩家位置进/出各记一份。

```
BunkerWorld.tscn  (Node2D, 独立场景)
├── Ground         TileMapLayer   (tilemap_bunker)
├── Collision      TileMapLayer   (tilemap_bunker_collision, layer 2)
├── Props          Node2D (y_sort)
│   ├── BunkerCrate×N   (装饰板条箱 t792)
│   ├── BunkerLootbox×N (可搜刮 t793, loot_type=51/52 → 05)
│   └── BunkerDoor×N    (房间隔断门, 复用 InteractionArea)
├── ZedSpawns      Node2D         # → 07 刷新点 (t791)
├── Exit           BunkerExit     (Area2D, 碰撞→返回地表)
└── PlayerSpawn    Marker2D       (进入落点)
```

```
BunkerEntrance.tscn  (地表入口, Node2D)
├── Sprite     Sprite2D       (bunker_entrance)
├── Body       StaticBody2D   (可选碰撞/装饰)
└── EnterArea  Area2D         (玩家进入→弹确认→切到 BunkerWorld)
```

**场景切换流程（idiomatic）**：
- `EventSystem`（或一个 `BunkerManager` autoload）持有「玩家在地表的坐标、地表场景引用、`depth`」。
- 进入：`EnterArea` 发 `enter_bunker(depth)` 信号 → 弹确认 UI → 淡入 → 隐藏/挂起地表 → 实例化 `BunkerWorld`（按 depth 选布局）→ 玩家移到 `PlayerSpawn`。
- 退出：`Exit` 发 `exit_bunker` → 淡入 → 销毁 `BunkerWorld` → 恢复地表 + 玩家回入口处。
- `inBunker` 状态存进 `EventSystem`，影响昼夜/天气/被动 XP（→ 09/12 存档）。

> 内部 loot（`BunkerLootbox` 的 `loot_type=51/52`）的具体物品表由 **05-战利品刷新系统** 拥有；本文只摆锚点 + 标池号。地堡僵尸由 **07** 拥有刷新逻辑。

## 3. 空投（组 `Airdrop`）

### 3.1 触发模型（✅ `game_events.decoded.txt:26311-26336`, 8573, 1103-1110）

原版空投是 **玩家主动请求**，不是定时自动落：

- HUD 上有一个**空投请求按钮**（`t706`，带 `ready2` 就绪动画）+ 冷却条（`t708`）。
- 冷却时长 `Airdrop_recharge` 按难度（场景开始设定）：

| 难度 | 空投冷却（秒） |
|---|:---:|
| Novice | **300** |
| Regular | **600** |
| Veteran | **900** |

- 玩家点请求 → 飞机 `t705`（Bullet 行为，`SetAngleOfMotion(90)` 向下飞）从地图顶飞入，飞到目标点**速度归零**（`CompareSpeed ≤ 0`）时：吐出空投箱 `t704` + 信号烟雾 `t719`，飞机销毁。
- 空投箱 `t704` 落地后启 **300s 计时器**（`StartTimer(300)`）——未拾取自动消失。
- 玩家碰到满不透明度的 `t704`（`CompareOpacity = 100`）→ 触发**单件分层抽取**开箱（见 3.2）。
- `t704.iv[0]` 决定外观帧（普通=0/帧切换；定向模式 5/10/11/12/20/21/22 用特殊帧）。

> **本项目触发模型（决策见 §决策）**：保留「请求按钮 + 冷却」语义，挂在 `Minimap`/HUD 上；落点用**玩家当前位置**（原版飞机飞向请求时的目标）。`EventSystem` 持冷却计时与就绪状态。

### 3.2 普通空投分层战利品（✅ 空投战利品概率表.md §2）

开箱抽**单件**，分层（每层 Else 嵌套）。**本表归 05 拥有**，此处只引用结构：

1. **第一层 基础食物**：`choose(1,2)=1`（**50%**）→ 基础食物池（罐头/饮料/蔬果）。
2. 否则进分级（难度越高越易掉到更稀有层）：
   - **中级**：`floor(random(50)) < 40/39/38`（N/R/V）命中 → 中级池（绷带/弹药/刀/暖宝宝…）。
   - **高级**：否则 `floor(random(11)) < 10/9/8` 命中 → 高级池（手枪/外套/背心…，满弹 iv=100）。
   - **顶级**：以上都未命中 → 顶级池（步枪/大背包/夜视/防弹衣…）。

> 完整物品池见 `数值探索/空投战利品概率表.md` 与 **05**。本文负责把「玩家碰箱 → 调用 05 的空投抽取函数」接起来。

### 3.3 定向空投（Seed-run / `SEED_TD=1`，→ 12）

「目标空投/Target Drop」种子模式：按箱子 `iv[0]` **定向投放固定组合**（不随机分层）：

| `iv[0]` | 内容 |
|:--:|---|
| 10 | 铁丝网×2 + 地雷×2 + 特殊弹×2 |
| 11 | 血袋 + 生理盐水 + 能量饮料 + 缝纫套件 |
| 12 | 瞄具 + 弹药×2 + 缝纫 + 清洁套件 |
| 20 | 枪+弹组合（`randy=choose(1..6)`，含 mac10 解锁分支） |
| 21/22 | 高级/顶级武器组（含解锁限定枪，**子表待核实**） |

> 定向投放绑定在 **种子挑战模式**（→ **12-难度与进度/种子**拥有 seed-run 规则）。本文实现普通空投，定向空投留 12 的种子模式接入接口（`AirdropCrate.target_drop_id`）。

### 3.4 本项目场景

```
AirdropPlane.tscn  (Node2D)
├── Sprite   Sprite2D
└── (脚本: 沿 angle 直线移动, 到达目标点 → emit dropped, queue_free)

AirdropCrate.tscn  (Node2D, y_sort)
├── Sprite       Sprite2D        (airdrop_box, frame 由 target_drop_id 决定)
├── Body         StaticBody2D    (落地后碰撞, 可选)
├── PickupArea   Area2D          (玩家进入→开箱)
```

## 5. `EventSystem`（autoload，调度 + 状态）

一个自动加载，集中持有这些事件的**运行期状态**，供 HUD/小地图/存档读取：

```gdscript
# event_system.gd  (autoload)
extends Node

signal airdrop_ready
signal airdrop_requested(pos: Vector2)
signal bunker_entered(depth: int)
signal bunker_exited()

var airdrop_recharge: float = 600.0   # 按难度 300/600/900 (→ 12)
var airdrop_cooldown: float = 0.0     # 剩余冷却
var in_bunker: bool = false
var bunker_depth: int = 0             # 供 09 被动 XP 速率读取 (待核实倍率)
var special_points: Array = []        # [{type, world_pos}] 供小地图画图标

func _process(delta):
    if airdrop_cooldown > 0:
        airdrop_cooldown -= delta
        if airdrop_cooldown <= 0:
            airdrop_ready.emit()

func request_airdrop(player_pos: Vector2) -> void:
    if airdrop_cooldown > 0: return
    airdrop_requested.emit(player_pos)   # AirdropSystem 实例化飞机
    airdrop_cooldown = airdrop_recharge
```

职责分工：
- **空投**：拥有冷却计时、请求入口；实例化飞机/箱子交给一个 `AirdropSystem` 子节点或本体方法。
- **地堡**：持 `in_bunker`/`bunker_depth`，协调场景切换（或委托 `BunkerManager`）；`bunker_depth` 暴露给 `09` 算被动 XP。
- **残骸/特殊点**：`special_points` 列表喂给小地图。
- **存档**（→ 12）：`airdrop_cooldown`、`in_bunker`、`bunker_depth`、已开过的残骸/地堡入口 占用集合，进存档。
- 难度参数（冷却 300/600/900、残骸/地堡数量表）由 `12` 的难度/关卡设定注入。

## 7. 决策

### 已定 ✅
1. **架构**：残骸/入口在世界生成时一次性实例化（沿用 `01` 不做距离 create/erase）；空投箱是运行期事件物；地堡是独立场景。
2. **残骸生成**：按 §1.1 关卡数量表 + 候选 block 落子，**先残骸后地堡**共用占用集合（复刻原版 `iv[9]` 避让），**one-shot 不重生**。
3. **空投触发模型**：保留原版「**玩家请求按钮 + 难度冷却**」（300/600/900），落点 = 玩家当前位置；非自动定时。`EventSystem` 持冷却。
4. **loot 归属**：残骸 loot（池 15）、地堡箱（池 51/52）、空投分层表全部由 **05-战利品刷新系统** 拥有；本文只摆锚点 + 标池号。
5. **调度**：用一个 **`EventSystem` autoload** 集中事件状态（冷却/`in_bunker`/`bunker_depth`/特殊点列表）+ 信号，存档进 `12`。
6. **小地图**：用 **`Control` HUD + `_draw()` 自绘**，读 `grid[16][16]`，道路支线帧复用 `01 §B5` 邻接逻辑；不照搬「平铺 Sprite + Pin」。
